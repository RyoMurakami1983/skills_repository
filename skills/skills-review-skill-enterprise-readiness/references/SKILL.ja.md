# エンタープライズ対応レビュー

チームや組織にスキルを展開する前に、エンタープライズグレードの要件を満たしているかを評価するワークフロー。個別の品質検証を超えて、ガバナンス、セキュリティ、テスト、運用安定性、コンプライアンスをカバーします。

## このスキルを使う場面

- チーム・部門・組織へスキルを展開する時（個人利用ではなく）
- コンプライアンス要件のある本番コードベースでスキルを使用する時
- サードパーティやコミュニティ提供のスキルを採用前にレビューする時
- スキルをチーム横断で標準化する準備をする時
- 既に展開済みのスキルを企業ポリシー準拠で監査する時

使わない場面：
- 新規スキルの執筆 → `skills-author-skill`
- 個別スキルの品質検証 → `skills-validate-skill`
- 個人・実験目的のみの場合

## Core Principles

1. **Defense in Depth（多層防御）** — 入力処理、出力生成、認証情報管理、依存関係の信頼性をすべてのレイヤーでレビュー（基礎と型）
2. **Governance as Enablement（ガバナンスは推進力）** — ガバナンスプロセスは安全な採用を加速すべきもの。明確な基準が曖昧さを減らし承認を速める（成長の複利）
3. **Operational Transparency（運用の透明性）** — チーム環境ではすべてのスキルが観測可能・監査可能・ロールバック可能であるべき（継続は力）
4. **Zero Trust for AI Output（AI出力をゼロトラスト）** — AI生成コードや指示を安全と仮定しない。実行前に検証・サンドボックス・レビュー（基礎と型 / ニュートラル）
5. **Compliance by Design（設計段階からコンプライアンス）** — コンプライアンスを後付けではなく、スキル構造に最初から組み込む（温故知新）

**なぜこの原則か**: 個人開発では品質検証だけで十分ですが、チーム・組織展開ではセキュリティ事故が複数人に波及し、コンプライアンス違反が法的リスクになります。「基礎と型」の精神で、展開前の型（チェックリスト）を定め、「成長の複利」としてチーム全体の安全性を底上げします。

## ワークフロー: エンタープライズ対応レビュー

### Step 1 — 前提条件の確認

```
事前チェックリスト:
├─ [ ] skills-validate-skillのレポートが存在し合格（Critical所見なし）
├─ [ ] SKILL.mdが1-skill-1-workflowの構造に従っている
├─ [ ] YAMLフロントマターにauthorフィールドが設定されている
├─ [ ] バイリンガル版が存在（SKILL.md + references/SKILL.ja.md）
└─ [ ] VersionとChangelogが存在する
```

### Step 2 — セキュリティレビュー

**2a. コード生成の安全性**
- ハードコードされた認証情報・トークン・シークレットがないか
- セキュリティ機能（SSL検証、CORS、認証）の無効化を指示していないか
- 生成コードに入力バリデーションの指針が含まれるか
- SQL/コマンドインジェクションのリスクが例示で対処されているか
- ファイルシステム操作に安全なパス処理が使われているか

**2b. 依存関係の信頼性**
- 参照パッケージ/ライブラリが信頼できるソースからか
- バージョン固定が推奨されているか（"latest"ではなく）
- 不必要な昇格権限を要求していないか

**2c. データ取り扱い**
- 例やテンプレートにPII/機密データがないか
- ログ出力の指針が機密フィールドを除外しているか
- エラーメッセージが内部詳細を漏洩しないか

### Step 3 — ガバナンスレビュー

**3a. オーナーシップと説明責任**
- 作者/オーナーが明確に特定されているか
- 保守責任が割り当てられているか
- 変更承認プロセスが定義されているか
- 廃止/引退計画が存在するか
- ライセンス互換性が確認済みか

**3b. スコープと境界**
- スキルのスコープが明確に限定されているか
- クロススキル依存が関連スキルに文書化されているか
- 既存チームスキルとの機能重複がないか
- 対象ユーザーが定義されているか

### Step 4 — テストと信頼性レビュー

- ワークフローのステップが再現可能か
- エッジケースが文書化されているか
- エラーハンドリングの指針があるか（ハッピーパスだけでなく）
- スキルが悪い出力を生成した場合のロールバック手順があるか
- スクリプトファイル（あれば）がターゲットプラットフォームでテスト済みか
- 例がそのままコンパイル/実行できるか

### Step 5 — 運用レディネスレビュー

- デプロイ方法が文書化されているか（グローバル？リポジトリレベル？）
- アップデート/アップグレードパスが明確か
- 問題発生時の検知方法があるか
- フィードバックチャネルが存在するか
- 新メンバーがすぐに理解できるか

### Step 6 — レポート生成

判定基準：
- **APPROVED**: 全セクションPASS、重大な所見なし
- **CONDITIONAL**: 軽微な所見あり、文書化された緩和策で展開可
- **REJECTED**: 重大なセキュリティ/ガバナンス所見あり、修正必須

## Good Practices

1. **展開前にレビュー、展開後ではなく** — エンタープライズレビューはゲートであり、事後検証ではない（基礎と型）
2. **繰り返しチェックは自動化** — セキュリティ・構造チェックはスクリプト化し、人間のレビューは判断が必要な部分に集中（成長の複利）
3. **展開スキルはバージョン固定** — エンタープライズ環境では自動更新せず、バージョンを固定（継続は力）
4. **承認済みスキルレジストリを維持** — どのスキルが承認/条件付き/却下かを追跡（ニュートラル）
5. **メジャー更新時は再レビュー** — 破壊的変更やセキュリティ関連の更新は再レビューをトリガー（温故知新）

## リスク分類による段階的レビュー

すべてのスキルに全項目を適用する必要はありません：

| 展開スコープ | レビュー範囲 | 項目数 |
|-------------|-------------|--------|
| 個人利用 | エンタープライズレビュー不要 | 0 |
| チーム | 軽量レビュー（Step 1, 2a, 3b） | ~13 |
| 組織全体 | フルレビュー（Step 1-6） | 34 |
| 外部公開 | フルレビュー + 法務/ライセンスチェック | 34+ |

**なぜ段階的か**: 「成長の複利」の精神で、プロセスの重さで採用をブロックしない。リスクに応じた適切な投資が、チーム全体のスキル活用を促進します。
