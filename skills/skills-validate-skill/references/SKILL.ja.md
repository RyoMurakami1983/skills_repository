# スキルを検証する

SKILL.mdファイルに対して品質検証を実行し、構造・コンテンツ・コード品質・言語の各カテゴリでスコア付きレポートを生成し、具体的な改善推奨を提供するワークフロー。

## When to Use This Skill

このスキルを使用する場面：
- 完成したSKILL.mdを公開・マージ前にレビューする時
- 構造化チェックリスト基準に対してスキル品質を評価する時
- スコアと具体的な改善項目を含む品質レポートを生成する時
- 「1スキル＝1ワークフロー」標準への準拠を確認する時
- 前回レポートの修正適用後に再検証する時

---

## Core Principles

1. **定量的評価** — 主観的な印象ではなく、客観的な合否基準を持つチェックリストを使用（基礎と型）
2. **カテゴリ別スコアリング** — 構造、コンテンツ、コード品質、言語を独立して評価（ニュートラル）
3. **実行可能なフィードバック** — 不合格項目にはすべて具体的で実装可能な修正推奨を含める（成長の複利）
4. **継続的改善** — 初回は≥80%を目標; 反復して90%+を達成（継続は力）

---

## ワークフロー：スキルを検証する

### ステップ1 — 構造検証

ファイル構造とセクション順序を確認。他の全チェックの前提条件。

**Why**: 構造が基礎です。基礎が揺らいでいる上に応用を積んでも、必ず崩れる。これが基礎と型の追求です。

**主要チェック項目**:
- YAML frontmatter完備（name, description, author, tags, invocable）
- `name:`がディレクトリ名と一致（kebab-case）
- "When to Use"が最初のH2
- 単一`## Workflow:`セクション（複数`## Pattern N:`ではない）
- ファイル長 ≤ 500行（550行まで許容）

### ステップ2 — コンテンツ検証

完全性、明確性、実用性を評価。

**Why**: コンテンツの質は「誰もが使える形式知」になっているかどうか。ニュートラルな視点で、暗黙の前提を排除し明示的に。

**主要チェック項目**:
- "When to Use"に5〜8個の動詞先導シナリオ
- Core Principlesが≥2個のPHILOSOPHY Valuesを参照
- ワークフローに順序立てたステップと例
- Good PracticesにValuesリンク

### ステップ3 — コード品質検証

コード例の品質と教育的価値を検証。

**Why**: コード例は「型」の実践。✅/❌マーカーでパターンを明確にし、WHYをコメントで伝えることが成長の複利。

### ステップ4 — 言語検証

文体と可読性を検証。能動態、短い文、一貫した用語。

### ステップ5 — レポート生成

構造化レポートにまとめる。各不合格項目に具体的な修正方法を記載。

### ステップ6 — 修正後の再検証

修正適用後、完全な検証を再実行して退行がないことを確認。

## Good Practices

### 1. 早期・頻繁に検証

**What**: 最後だけでなく、執筆中に検証を実行。
**Why**: 問題を早期に発見することで複合エラーを防止。
**Values**: 継続は力

### 2. 構造を最初に修正

**What**: 構造の不合格は他のすべてをブロック。
**Why**: 基礎が正しくなければ、コンテンツや品質の評価は意味がない。
**Values**: 基礎と型

### 3. レポートを学習資産として活用

**What**: 品質レポートを保存し、共通の失敗パターンを分析。
**Why**: レポートが将来のスキル著者のための複利的学習資料になる。
**Values**: 成長の複利

---

## Common Pitfalls

### 1. クリティカル不合格があるのにスキルを公開

**Problem**: 全体85%でもクリティカル構造不合格がある。
**Solution**: クリティカル項目はハードゲート — 全体スコアに関係なく全てパス必須。

### 2. 修正後に再検証しない

**Problem**: 1つの問題を修正して別の問題を導入（退行）。
**Solution**: 変更後は常に完全な検証を再実行。

## Quick Reference

### 重要度レベル

| 重要度 | 意味 | アクション |
|-------|------|----------|
| Critical | 公開をブロック | 即時修正必須 |
| Error | 品質に大きく影響 | 公開前に修正 |
| Warning | 軽微な品質問題 | 90%+達成に修正 |
| Bonus | 追加品質ポイント | あれば良い |
